CatMouse vs. MouseBoard

Overview

What kind of game would you make if you had one week to make a local multiplayer game that only used a mouse and keyboard? My answer was a literal game of Cat and Mouse. One player uses a computer mouse to control the Cat while another uses a keyboard to control the Mouse.

Lessons From Project

One of the biggest lessons from this project was the difficulty of different input options. Balancing the Cat and Mouse was an interesting challenge since the Cat intuitively has the upper hand. Initially this lead to the Cat winning almost every round! My first thoughts were to just nerf the Cat in some way, but a comment from a fellow designer made me realize that smaller changes could have the same impact without compromising the asymmetrical multiplayer experience. A combination of timers, limitations, and visual communication has allowed the game to feel fair for both players. While the Mouse is still a harder character to control input-wise, overall the game still feels fun for both players.

Design Process Summary

CatMouse vs MouseBoard started as a challenge to make a local multiplayer game with no dedicated controllers but quickly evolved into something more. Various parts of the project completely changed the game until it became the product you see here. Some of those design decisions include:
    •Established a design credo early to keep the overall vision consistent throughout the project •Changed the controls from a menmonic scheme to one that created a unique experience •Balanced gameplay based on a variety of factors including input method intuitiveness

Project Goals

  • •Create an original game that fit the challenge theme
  • •Improve programming skills by building unorthodox gameplay
  • •Recapture the magic from childhood memories of playing local multiplayer games

Initial Idealization

The confines of the challenge were the initial building blocks for the idea. I thought back on my own local multiplayer experiences. Like how my brother and me would have to crowd around a single keyboard, I decided that I’d use the mouse too to save players from the same experience. These vastly different input methods informed the asymmetrical multiplayer design, but not in the ways I thought it would.



Balancing an Asymetrical Multiplayer Game

The original concept had the Mouse controlled by the mouse, which made mnemonic sense but gameplay wise not so much. The Mouse could easily avoid the Cat, even when the controls were just using the keyboard for movement and swiping. It just wasn’t fun for either player, so I went back to the drawing board. I looked at other pieces of media that played up the cat vs. mouse formula and got inspired by the most classic example. Almost every episode of Tom and Jerry starts out with the mouse hiding in a wall.


Player Experience Goals

The main goal was to recreate the feeling of playing against a sibling or friend as a kid. Different roles for both players allows for instant replayability. Shorter play times and a persistent score for each play session entices players to be more competitive. Other small details were added to make the game more playful, like letting the cat player meow on right click. Three different sounds would play, two normal cat meows and one of me saying it in a deep voice. The last one was added to delight any player who was already feeling whimsical enough to hit a input that has no effect on the gameplay.



Postmortem

Since this project was made on a time crunch, I didn’t have time to add details beyond making sure it conveyed the concept. The sprite art is from the first pass since it was a lower priority for the project, but ultimately needs to be more stylized to emphasize the player experience goal. Balancing needs to be fine-tuned even more, with more formal playtests to pinpoint what needs the most tweaking. Finally this is a small detail, but I should’ve given the Mouse a way to squeak. I hope that by adding something like this can get players into the mindset of competitive silliness.
